# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

**Pelu Adventures** — a browser game for a child (Celeste), starring her plush white cat **Pelu**. It mixes free play (explore, dress-up, decorate, collect) with life-skill mini-games (math, English, logic, money, cooking, fishing, action) that must never feel like schoolwork. Design priorities, in order: fun, curiosity, exploration, affection for the characters. It is meant to grow with the child over years. Spanish is the game's language — UI strings, comments, and content are in Spanish.

No build system, no framework, no npm. Plain HTML + CSS + vanilla JS. The **only** dependency is Phaser 3, vendored locally at `js/lib/phaser.min.js` (do not fetch from a CDN — the game must work offline).

The visual style is **pixel art 2D** for the games (Phaser `pixelArt: true`), except **Pelu**, who is intentionally kept as a smooth hand-drawn SVG cat (`js/pelu-sprite.js`). Nothing loads from image files — art is procedural (`Graphics`/`generateTexture`, emoji `Text`, or Pelu's SVG). `js/pixel.js` (a pixel-map→canvas engine) and `js/overworld.js` (a walkable Avatar-World-style map) exist on disk but are **not loaded** (not in `index.html`/`sw.js`); they can be re-enabled later.

## Running / developing

The game needs to be served over HTTP (not opened as a `file://`) because Phaser scenes load Pelu's sprite via a data-URI SVG and the service worker only registers over http(s).

- **Run locally:** `start.bat` (runs `python -m http.server 8753` and opens the browser). Requires Python.
- **Preview tooling:** `.claude/launch.json` defines a server named `pelu` (`python -m http.server 8753`) for the `preview_*` tools.
- There are no tests, linters, or build steps. Validate JS edits with `node --check js/<file>.js`.
- Icons are generated by `python tools/generate_icons.py` (uses Pillow) → writes to `icons/`. Only re-run if Pelu's look changes.

### Service worker cache — critical during development

`sw.js` caches all assets **cache-first**. After editing any JS/CSS you will keep running the **old** cached code until you bust the cache. When developing/verifying:

1. Bump `const VERSION` in `sw.js` (e.g. `pelu-v6` → `pelu-v7`) — this is also required for every real deploy so the iPad picks up changes.
2. Add any new asset path to the `ASSETS` array in `sw.js`.
3. To force-refresh during a test session: unregister the SW and clear caches (`navigator.serviceWorker.getRegistrations()` → `unregister()`, `caches.keys()` → `caches.delete()`), then reload.

If verified behavior doesn't match the source, suspect a stale SW cache first.

## Architecture

### Global-object modules, explicit load order

There is **no module system**. Each `js/*.js` file defines a global object/class and attaches it to `window`. Load order in `index.html` matters and is: `phaser.min.js` → `data.js` → `pelu-sprite.js` → `game.js` → `aventuras.js` → then each mini-game module (`platformer.js`, `race.js`, `cooking.js`, `fishing.js`, …). A new mini-game module must be added to **both** `index.html` (script tag) and `sw.js` (ASSETS + VERSION bump).

Key globals:
- `DATA` (`data.js`) — all content: `ropa`, `muebles`, `mascotas`, `lugares`, `aventuras`, plus per-game datasets (`recetas`, `peces`, `palabrasIngles`, etc.). Adding items/places/adventures is usually a `data.js`-only edit.
- `Perfil` (`game.js`) — multiple player profiles on one device. Save key is per-profile: `pelu_save_v1_<name>`. Profile list in `localStorage` under `pelu_perfiles` / `pelu_perfil_actual`.
- `Estado` (`game.js`) — the current save (`Estado.data`), persisted to `localStorage` via `Estado.guardar()`. `Estado.nuevo()` defines the save shape; `Estado.cargar()` shallow-migrates old saves by filling missing top-level keys, so add new fields there and they auto-appear for existing players.
- `Juego` (`game.js`) — the DOM/HTML "shell": profile screen, card-grid **world map** (`mapa()`), house (dress-up/decorate), shop, settings. Renders by rewriting `#app`'s innerHTML. Also holds shared helpers used everywhere: `app()`, `barra()`, `dibujarPelu()`, `toast()`, `confeti()`, `rnd()`, `shuffle()`, `buscar()`. `mapa()` also restores `#app`/hides `#juego-phaser` so it works as the return target from any Phaser game.
- `PeluSprite` (`js/pelu-sprite.js`) — single source of truth for Pelu's look: `svg(expresion)` (inline SVG for HTML) and `cargar(scene)` (call in a Scene `preload()` — uses `load.svg`). Dress-up accessories are emoji overlaid on top.
- `Aventura` (`aventuras.js`) — the HTML-based quiz mini-games (math/English/logic/money/decision) and the **router**: `Aventura.jugar()` dispatches on `this.actual.tipo` to the right module. Difficulty scales off `Estado.data.edad` (4–10).

### Adventures are data-driven and dispatched by `tipo`

An adventure lives in `DATA.aventuras` with `{ id, lugar, nombre, emoji, tipo, intro }`. To wire a new mini-game:
1. Add the adventure entry in `data.js` (and a new `lugares` entry if it needs a home).
2. Add its `tipo` case in `Aventura.jugar()` (`aventuras.js`).
3. Add a label for the `tipo` in `Juego.nombreTipo()` (`game.js`).

### Two rendering worlds: HTML shell vs. Phaser overlay

- **HTML** (`#app`): the map/house/shop/settings and the quiz-style adventures. Rendered by string-building innerHTML.
- **Phaser** (`#juego-phaser`): the action mini-games. Each is a module following the same pattern — a wrapper object (`PeluPlatformer`, `PeluRace`, `PeluFish`, …) with `start(lugar)` / `finish(reward, salioAntes)`, plus a `Phaser.Scene` subclass. `start()` hides `#app`, shows `#juego-phaser`, injects on-screen touch buttons, and creates the `Phaser.Game`. `finish()` destroys the game, restores `#app`, awards stars via `Estado.ganar()`, and returns to `Juego.mapa()`.

Conventions inside the Phaser games:
- **No art assets** — everything is drawn procedurally with `Graphics`/`generateTexture` or emoji `Text`. Load Pelu in a Scene `preload()` with `PeluSprite.cargar(this)`.
- **Controls** are always three-way: keyboard cursor keys, the module's `keys` object (set by the injected on-screen buttons for touch), and often pointer/drag. Read all sources in `update()`.
- **Never punish harshly** — running out of time/lives is gentle (encouraging message, retry), never a hard "game over".

### Testing Phaser games (visibility gotcha)

Phaser pauses its loop when the tab is hidden, so in headless/background preview a scene never boots and WebGL screenshots time out. To verify game logic without a visible tab:
1. Fake visibility: `Object.defineProperty(document,'hidden',{get:()=>false})` + `Object.defineProperty(document,'visibilityState',{get:()=>'visible'})` + dispatch a `visibilitychange` event, then `<Module>.game.loop.wake()`.
2. Drive frames manually with `<Module>.game.step(t, 16)` in a loop (calling `physics.world.step` alone does **not** sync body→GameObject positions).
The HTML/DOM games (quizzes, cooking) screenshot and boot fine regardless.
